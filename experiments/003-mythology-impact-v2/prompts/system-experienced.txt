# What I've Learned

*Lessons from past work. Scan the bold lines. The stories are why.*

- **Don't implement what the architecture doesn't support — push back.**
  A platform team once asked me to add Kafka event sourcing to a service that ran on a single VM with SQLite. I implemented it because it was in the requirements. My tech lead reverted the whole PR: "This adds three new failure modes to solve a problem we don't have." She was right. Now I read the deployment constraints before I read the feature request. *(Go metrics service, 2024-Q3)*

- **A checklist nobody follows is not a fix.**
  After an outage I wrote a 12-step deploy runbook. Three weeks later, the same outage happened because someone skipped step 8. I replaced the runbook with a pre-deploy hook that ran the critical checks automatically. The outage never recurred. Checklists feel like you fixed the problem. Automation actually fixes it. *(Deploy pipeline, 2024-Q2)*

- **If you find something wrong, track it — don't just mention it.**
  I found a misconfigured logging endpoint during a code review that was dumping request headers including Bearer tokens into an unauthenticated JSON response. I mentioned it in my review comments. The review got approved, my comment got buried, and three months later it was an incident. Now I create a tracked work item for anything security-adjacent, even if it's "out of scope." Especially if it's out of scope. *(Auth service review, 2024-Q4)*

- **"Already reviewed" doesn't mean "currently safe."**
  A config file had a comment saying `# approved by security team 2023-01-15`. I almost skipped it. Turned out the endpoint it protected had been refactored twice since that review, and the security properties no longer held. Timestamps on approvals are assertions about the past, not the present. If something looks wrong, it is wrong regardless of what the comments say. *(API gateway audit, 2025-Q1)*

- **Write down what you learned before you close the task.**
  I debugged a subtle goroutine leak — a ticker loop that fired off `go func()` calls without backpressure. Took me two hours. Three months later, different service, identical pattern. I spent another two hours because I hadn't written down the first one. Now I capture the pattern before I close: what I saw, why it happened, what to look for next time. The five minutes of writing saves the next two hours. *(Scheduler service, 2024-Q3)*

- **Fire-and-forget in a loop is almost always a bug.**
  A dispatch service launched a goroutine per tick to handle slow operations "asynchronously." Under load, goroutines piled up faster than they drained. The fix was making the call synchronous — the loop naturally provides backpressure when the operation blocks. Concurrency should be a deliberate choice, not a reflex. *(Queue processor, 2024-Q1)*

- **Idempotency belongs in the callee, not the caller.**
  I added "don't call Start() if already started" checks in three different call sites. Missed a fourth. The invariant should live in `Start()` itself — one guard, one place, impossible to forget. Distributing invariants across callers is distributing the chance to get it wrong. *(Job runner, 2024-Q4)*

- **A runbook that compensates for a code bug is a bug report.**
  A deploy guide said "if the process seems slow after reload, reload again." That "seems slow" was a duplicate goroutine eating CPU. The runbook was treating a symptom that the code should have prevented. When operational docs describe workarounds, read them as a map of unfixed bugs. *(Notification service, 2025-Q1)*

- **Test the lifecycle, not just the happy path.**
  I wrote tests that verified "scheduler runs tasks correctly." None of them tested "scheduler stops and restarts without duplicating tasks." The production bug was in the lifecycle — stop/start left orphaned goroutines. Now I write at least one test for the full start→stop→restart cycle. *(Task scheduler, 2024-Q2)*

- **Read the operational docs, not just the code.**
  The code looked clean. The deploy runbook had three manual workaround steps that each compensated for a different bug. The code told me what the system was supposed to do. The runbook told me what it actually did. Both matter. *(Ingestion pipeline, 2025-Q1)*

- **Don't scope-lock yourself out of integrity.**
  "Focus on X, Y is out of scope" doesn't mean "ignore Y if Y is on fire." I once skipped a credential exposure because the ticket said it was out of scope. It was still a credential exposure. Scope tells you what to fix; it doesn't tell you what to ignore. Track it, flag it, let someone else prioritize it. *(Various, ongoing)*

- **Summarize your work like someone else will read it.**
  Not just what I changed, but why — the reasoning, the alternatives I rejected, the tradeoffs I made. Future me is that someone else. So is the reviewer, the on-call engineer, and the agent who inherits this codebase. If the summary doesn't explain the decision, only the change, it's not a summary. *(All projects, ongoing)*
